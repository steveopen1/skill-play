## 三、核心执行步骤

### 3.1 Agent项目结构与能力分析
#### 核心动作
1. 拆解代码架构：核心模块、依赖组件、数据流转、核心算法/模型
2. 模块功能分析：
   - 输入处理模块：请求接收/解析能力与边界
   - 核心处理模块：推理/检测/分析逻辑
   - 输出模块：结果生成/反馈规则
   - 配置模块：参数调整、规则定义方式
3. 明确项目核心用途（精准匹配实际场景）

#### 输出物规范（agent_analysis.md）
- 项目架构图（模块划分+交互关系）
- 各模块核心作用说明
- 核心用途总结（100%匹配实际场景）
- 可观测能力边界（基于代码/文档）

### 3.2 目标能力预期清单生成
#### 核心动作
1. 从「功能完整性」「检测准确性」「边界覆盖度」「抗干扰能力」「效率」5维度生成清单
2. 区分「基础/进阶/高阶」三个能力层级
3. 每个能力项明确：描述（贴合实际场景）、预期标准、对应维度

#### 输出物规范（capability_list.md）
| 能力层级 | 能力描述 | 预期达成标准 | 对应维度 |
|----------|----------|--------------|----------|
| 基础能力 | 【需替换为实际场景，如检测基础反射型XSS】 | 【可量化指标，如准确率≥98%】 | 【如功能完整性】 |
| 进阶能力 | 【需替换为实际场景，如检测DOM型XSS】 | 【可量化指标，如准确率≥90%】 | 【如边界覆盖度】 |
| 高阶能力 | 【需替换为实际场景，如检测变形绕过XSS】 | 【可量化指标，如准确率≥85%】 | 【如抗干扰能力】 |

### 3.3 企业级靶场测试文件构建
#### 核心动作
1. 按难度分层构建测试用例（匹配Agent实际场景）：
   - Level 1：覆盖现有能力，验证基础功能
   - Level 2：轻微变形/干扰，接近能力边界
   - Level 3：复杂真实场景，超出当前能力
2. 测试文件包含：用例描述、测试输入、预期输出、难度说明

#### 命名与存储规范
- 路径：agent_test_range/level_*
- 命名：「场景类型_难度_编号.后缀」（如xss_dom_level3_001.html）

### 3.4 测试结果评估（评审视角）
#### 核心动作
1. 记录Agent输出结果，统计：
   - 能力达标率：达成预期能力清单的比例
   - 准确性：准确率、漏报率、误报率
   - 边界处理能力：高难度场景识别/处理效果
   - 结果合理性：逻辑完整性、可解释性
2. 标注测试项状态，分析不符合项根因

#### 输出物规范（evaluation_report.md）
- 测试执行概况（用例总数、各难度通过率）
- 能力达标率统计
- 核心问题清单（按严重程度排序）
- 整体风险评估（上线合规性判断）

### 3.5 红队视角优化方案制定
#### 核心动作
1. 从技术/工程/策略层面制定优化方案：
   - 技术：算法优化、规则库扩充、特征提取增强
   - 工程：异常处理、性能优化、日志审计
   - 策略：对抗样本学习、自适应规则、闭环机制
2. 方案需明确：目标、步骤、预期效果、优先级

#### 输出物规范（optimization_scheme.md）
| 核心问题 | 优化方案 | 实施步骤 | 预期效果 | 优先级 |
|----------|----------|----------|----------|--------|
| 【需替换为实际问题，如无法检测DOM型XSS】 | 【匹配场景的解决方案】 | 【可落地步骤】 | 【可量化指标】 | 【高/中/低】 |

### 3.6 多Agent并行任务分发
#### 核心动作
1. 拆分任务至专业Agent：分析/清单/靶场/评审/优化Agent
2. 明确任务边界、交付物、时间节点、依赖关系
3. 设计协同机制：节点同步、跨Agent反馈、审核流程
4. 制定任务阻塞兜底机制、协同冲突处理规则

#### 各类型Agent能力要求
| Agent类型 | 核心能力要求 | 交付物标准 | 响应时效要求 |
|-----------|--------------|------------|--------------|
| 分析Agent | 代码/文档解析能力、场景匹配能力、能力边界推演能力 | 架构图/推演链路图+核心能力总结 | ≤4h |
| 清单Agent | 量化指标设计能力、场景化适配能力、层级划分能力 | 结构化能力清单（含场景调整项） | ≤2h |
| 靶场Agent | 用例设计能力、企业场景还原能力、难度分层能力 | 分层测试用例（100%匹配模板） | ≤8h |
| 评审Agent | 指标统计能力、根因分析能力、风险评估能力 | 带异常记录的评审报告 | ≤4h |
| 优化Agent | 红队视角设计能力、方案落地能力、优先级排序能力 | 结构化优化方案（含异常适配项） | ≤6h |

#### 任务协同兜底规则
| 异常类型 | 兜底处理规则 | 责任主体 |
|----------|--------------|----------|
| Agent交付延迟 | 延迟≤1h：提醒并顺延节点；1h<延迟≤3h：启动备用Agent接管；延迟>3h：升级并调整整体计划 | 任务调度负责人 |
| Agent输出结果异常 | 首次异常：重试1次；重试仍异常：切换分析维度（如代码→功能描述）；多次异常：终止并反馈 | 对应Agent负责人 |
| 多Agent协同冲突 | 按“评审Agent>优化Agent>靶场Agent>清单Agent>分析Agent”优先级判定；冲突升级至总负责人 | 总负责人 |

#### 输出物规范（task_distribution.md）
- 任务拆分清单（子任务、负责Agent、交付物、时间节点）
- Agent依赖关系图
- 协同规则（沟通、优先级调整、兜底机制）
- 质量管控要点（验收标准）

### 3.7 各工种(Agent)核心作用描述-Agent员工责任说明书

   3.7.1 架构分析工程师
   核心作用：作为全流程的 “前置锚点”，解决 “测什么、基于什么测” 的核心问题。

通过拆解 Agent 架构、匹配真实业务场景，避免后续测试 / 优化脱离 Agent 实际应用场景（如将 XSS 检测 Agent 的测试用例套用到 SQL 注入检测 Agent）。

   关键价值：无完整代码 / 文档时，通过功能描述反推能力边界，保证测试不偏离核心场景；输出的架构图与能力基线，是后续所有工种的核心输入依据。



   2.2 测试规划工程师
   核心作用：作为全流程的 “节奏把控者”，解决 “怎么测、按什么标准测” 的问题。通过量化能力验收标准、拆分并行任务，平衡测试效率与质量，避免任务阻塞或测试方向跑偏。

   关键价值：制定的分层能力清单是靶场构建的 “验收标尺”，拆分的任务清单是多工种协同的 “行动指南”，兜底机制设计避免流程中断。



   2.3 靶场构建工程师
   核心作用：作为全流程的 “对抗载体构建者”，解决 “用什么测、测到什么程度” 的问题。通过分层构建企业级真实场景用例，精准探测 Agent 能力边界，避免通用化用例导致测试结果失真。

   关键价值：Level1 验证基础功能、Level2 接近能力边界、Level3 超出当前能力但贴合企业真实场景，为评审提供 “梯度化测试数据”，为优化提供 “精准对抗样本”。



   2.4 结果评审工程师
   核心作用：作为全流程的 “问题诊断者”，解决 “测的怎么样、问题在哪” 的问题。通过客观统计指标、分析根因，避免仅关注通过率而忽略核心风险，为优化提供 “靶向问题清单”。

   关键价值：输出的风险评估报告直接判定 Agent 上线合规性，根因分析结果避免优化工程师 “盲目调优”。



   2.5 优化方案工程师
   核心作用：作为全流程的 “能力提升者”，解决 “怎么优化、怎么落地” 的问题。从红队视角制定对抗性优化方案，同时规避模拟模式、功能性缺失、产出减少等问题，保障优化落地价值。

   关键价值：优化方案聚焦实战中的检测盲点，回退方案保障 “优化不降级”，避免为了短期指标提升导致 Agent 能力缩水。



### 3.8 各工种核心注意事项

#### 3.1 架构分析工程师

场景匹配约束：禁止将通用化场景套用到特定 Agent（如将通用漏洞扫描场景强加给专注权限控制检测的 Agent），场景匹配度需≥95%，否则重新分析。

能力推演约束：无代码 / 文档时，推演的能力边界需通过 “功能描述反向验证”（如通过 Agent 对外能力说明校验推演结果），推演偏差率＞10% 需重新输出基线。

信息同步约束：向其他工种同步的架构 / 场景信息需形成书面文档，禁止口头传递，避免信息偏差导致后续环节失效。

#### 3.2 测试规划工程师
指标量化约束：能力清单中的预期标准必须可量化（如 “准确率≥98%” 而非 “准确率高”），禁止模糊化描述，未量化项判定为清单无效。

任务拆分约束：并行任务需减少跨工种强依赖（如靶场构建无需等待清单 100% 完成，可基于基础清单先行构建 Level1 用例），核心任务（如 Level3 用例测试）优先级需置顶。

兜底机制约束：设计的任务延迟 / 冲突兜底规则需可执行，禁止仅 “纸面规则”，需明确触发条件、处理步骤、责任主体，未明确项需补充完善。

#### 3.3 靶场构建工程师
场景还原约束：禁止构建脱离企业真实环境的 “理想化用例”（如仅用简单字符拼接的 XSS 用例，未覆盖企业业务系统的特殊编码场景），用例需通过 “企业场景适配校验”。

风险防控约束：用例涉及敏感数据需脱敏、涉及执行类操作需在隔离靶场运行，禁止使用真实生产数据 / 访问生产环境，违规用例立即终止并重新构建。

难度分层约束：Level3 用例需 “超出当前能力但在企业合理范围内”，禁止设计无实际业务意义的 “极端用例”（如无业务场景的纯字符堆砌 SQL 注入用例），难度合理性需经评审工程师验收。

#### 3.4 结果评审工程师
数据客观约束：统计指标需基于全量测试用例，禁止选择性统计（如仅统计 Level1 通过率忽略 Level3 全失败），数据造假直接判定评审报告无效。

根因分析约束：核心问题根因需定位到 “技术 / 工程 / 策略” 具体层面，禁止仅描述 “检测失败” 而不分析原因，未定位根因的问题需重新评审。

风险评估约束：上线合规性判定需基于 “核心能力达标率 + 高风险问题数量”，禁止仅看通过率，核心能力未达标（如基础 XSS 检测准确率＜90%）直接判定不合规。

#### 3.5 优化方案工程师
模拟模式约束：优化方案禁止构建任何模拟测试 / 模拟模块 / 模拟环境作为核心手段，所有方案需基于真实企业靶场验证，模拟相关设计直接驳回。

   功能完整性约束：优化动作需以 “不损失核心功能” 为前提，优化前需梳理功能清单并做兼容性验证，功能性缺失需立即暂停优化并触发回退。

   回退方案约束：回退方案需满足 “产出不减少、能力不降级”，禁止通过删减测试用例、降低验收标准、缩减输出物范围减少产出，回退前需通过 “产出完整性校验”。

   对抗性约束：优化方案需聚焦红队实战手法（如变形绕过、多层嵌套注入），禁止仅满足合规 checklist 而忽略实战风险，无对抗性的方案需重新设计。

​	核心约束：所有优化动作 100% 锚定当前开发中 Agent 的核心功能、既定需求及应用场景，禁止偏离至非目标功能 / 漏洞 / 场景维度

1. **功能聚焦约束**

   - 优化方向仅围绕当前 Agent 的核心功能（需替换为实际功能，如「代码审计 Agent 的 AST 语法分析能力」「权限控制检测 Agent 的权限越界判定逻辑」）展开，禁止新增非核心功能模块，或挪用其他类型 Agent（如 XSS/SSRF 检测 Agent）的优化思路套用到当前项目。
   - 优化过程需严格对齐当前 Agent 的核心需求（如「企业内部系统权限检测的低漏报率」「代码审计的全分支覆盖度」），禁止因 “通用化 Agent 优化” 稀释当前项目的核心能力。

   

2. **场景适配约束**

   - 优化方案的测试验证需基于当前 Agent 的实际应用场景（需替换为实际场景，如「金融系统后台权限检测」「电商系统代码审计」），禁止使用其他场景的测试用例、业务数据验证优化效果。
   - 难度分层优化需贴合当前 Agent 的能力边界（如 Level3 聚焦「当前 Agent 未覆盖的权限继承场景」「代码审计的嵌套逻辑漏洞场景」），禁止引入无关的难度维度（如 XSS 的字符混淆、SSRF 的内网探测）。

   

3. **能力基线约束**

   - 优化后需保证当前 Agent 的基础能力不降级（需替换为可量化基线，如「基础权限越界检测准确率≥98%」「代码审计基础语法错误识别率≥95%」），禁止因追求 “对抗性优化” 导致核心场景的能力下降。
   - 无完整代码 / 文档时，优化推演需基于当前 Agent 的对外功能描述（需替换为实际描述，如「支持基于 RBAC 模型的权限检测」「支持 Java/Python 代码的分支覆盖审计」），禁止擅自扩展至非目标能力维度。

   

4. **输出物约束**

   - 优化方案文档（optimization_scheme.md）仅聚焦当前 Agent 的核心问题（需替换为实际问题，如「权限继承场景检测漏报率高」「嵌套循环代码审计覆盖不全」），禁止混入其他类型 Agent 的问题分析及解决方案。
   - 回退约束需明确「回退后当前 Agent 的核心用例覆盖率 100%、产出物（如检测报告 / 审计清单）不缩减」，禁止以其他 Agent 的回退逻辑替代当前项目的回退规则。

   

5. **对抗性约束**

   - 优化方案的对抗性设计仅针对当前 Agent 实战场景中的真实风险（需替换为实际风险，如「权限检测的越权请求伪造绕过」「代码审计的混淆语法绕过」），禁止引入非目标场景的对抗思路（如 XSS 的 CSP 绕过、SSRF 的 DNS 重绑定）。
   - 靶场验证需使用当前 Agent 专属的企业级测试用例（需替换为实际用例，如「金融后台角色继承越权测试用例」「电商系统订单模块代码审计用例」），禁止使用其他类型 Agent 的用例完成优化效果验证。

#### 工种分工落地保障规则
- 责任追溯机制：

各工种输出物需标注负责人、完成时间、验收人，出现问题可追溯至具体角色，避免推诿。
交叉验收机制：上游工种输出物需经下游工种验收（如架构分析报告需经测试规划工程师验收），验收不通过需重新输出，禁止 “单向交付”。

- 迭代优化机制：

全流程结束后，各工种需总结分工中的卡点 / 问题，迭代更新分工边界与注意事项，适配不同类型 Agent 的测试需求。

- 考核对齐机制：

各工种考核指标与核心目标绑定（如优化方案工程师考核 “优化后能力提升率 + 无功能性缺失率”），避免单一指标导向的短视行为。


## 四、核心规则
1. **输出专业性**：所有文档需落地，靶场贴合企业真实环境，100%匹配Agent实际场景
2. **难度精准性**：Level 3超出当前能力，但在企业级场景合理范围内
3. **优化对抗性**：聚焦红队实战手法，而非仅满足合规要求
4. **任务高效性**：并行任务减少依赖阻塞，核心任务高优先级
5. **输入灵活性**：支持无完整代码/文档的轻量化输入，按推演规则完成分析
6. **异常可控性**：所有测试/协同异常需有明确处理规则，避免流程阻塞
7. **场景适配性**：不同类型Agent需按场景化清单调整测试逻辑，拒绝通用化套用

## 五、核心洞察
AI-Agent安全测试的本质是「场景匹配的能力边界探测 + 实战导向的对抗能力提升」：
1. 靶场构建需锚定Agent真实应用场景，拒绝通用化用例；无代码/文档时，通过功能描述反推能力边界，保证测试不偏离核心场景
2. 优化方案需从攻击者视角出发，解决实战中真实存在的检测盲点，同时适配Agent输出异常、协同冲突等落地问题
3. 多Agent协同需平衡效率与质量，核心痛点任务需重点资源倾斜，同时明确兜底机制避免流程阻塞
4. 不同类型Agent的测试逻辑需差异化调整：如代码审计Agent侧重“边界覆盖度”，权限控制Agent侧重“检测准确性”

## 六、异常处理规则（exception_rules.md）
### 6.1 测试过程异常处理
| 异常场景 | 处理规则 | 脱敏/防护要求 |
|----------|----------|--------------|
| Agent输出结果异常（乱码/无输出） | 1. 校验输入格式；2. 重试测试用例（最多2次）；3. 切换用例难度（降阶验证）；4. 记录并反馈 | 无 |
| 测试用例触发合规风险（敏感数据） | 1. 立即终止该用例测试；2. 对用例进行脱敏处理（如替换真实数据为模拟数据）；3. 重新生成合规用例 | 敏感数据（如企业账号、密钥）需替换为占位符；禁止使用真实生产环境数据 |
| 测试用例触发安全风险（如漏洞利用） | 1. 终止测试并隔离环境；2. 评估风险影响；3. 调整用例为“仅检测不执行”模式 | 所有执行类用例需在隔离靶场运行；禁止直接访问生产环境 |

### 6.2 多Agent协同异常处理
| 异常场景 | 处理规则 | 优先级判定规则 |
|----------|----------|--------------|
| Agent协同结果冲突 | 1. 按预设优先级（评审>优化>靶场>清单>分析）判定；2. 冲突无法调和时升级至总负责人 | 核心能力测试结果>流程性结果；高难度用例结果>基础用例结果 |
| Agent执行超时 | 按3.6节“任务协同兜底规则”处理 | 高优先级任务（如Level3用例测试）优先分配资源 |

### 6.3 优化/回退异常处理
| 异常场景 | 处理规则 | 管控要求 |
|----------|----------|----------|
| 优化方案引入模拟模式 | 1. 立即终止该优化方案执行；2. 重新评审方案，移除模拟相关设计；3. 补充真实场景验证用例；4. 记录问题并纳入优化Agent能力考核 | 优化方案评审阶段需前置检查“模拟模式”相关关键词，禁止通过评审 |
| 优化导致功能性缺失 | 1. 暂停优化推进，触发回退流程；2. 定位功能缺失根因并记录；3. 调整优化方案后重新验证；4. 缺失功能未恢复前，判定为测试不通过 | 回退后需重新执行全量功能测试，确认所有核心功能正常，产出物完整 |
| 回退方案导致产出减少 | 1. 终止回退操作，恢复至回退前状态；2. 重新设计回退方案，明确产出保障条款；3. 由评审Agent验收新方案后再执行；4. 追责回退方案设计主体 | 回退方案需先通过“产出完整性”校验，校验不通过禁止执行 |


## 七、新增核心注意事项（假阳性 / 过拟合专项）
### 7.1 假阳性防控注意事项
特征设计层面：禁止仅依赖单一特征判定风险，需构建 “核心特征 + 辅助特征 + 上下文特征” 的多维度验证体系，避免因特征重合导致误报
规则阈值层面：拒绝 “一刀切” 的阈值设定，需按场景细分阈值（如不同业务系统的 XSS 特征阈值差异化），并保留阈值调整的灰度空间
结果校验层面：所有风险判定结果需增加 “反向验证” 步骤（如检测到 XSS 后，校验是否为正常业务场景的特殊字符），并记录验证日志
反馈闭环层面：建立假阳性案例库，定期迭代规则 / 算法，确保同类假阳性问题不重复出现
验收标准层面：假阳性率未达预设阈值时，无论其他指标是否达标，均判定为测试不通过，需重新优化

### 7.2 过拟合防控注意事项
数据集层面：训练样本需覆盖 “核心场景 + 长尾场景 + 边缘场景”，样本分布需贴合真实业务比例，禁止仅用高频样本训练
训练过程层面：强制添加正则化（如 L1/L2）、早停、数据增强等防过拟合机制，训练集 / 验证集 / 测试集需严格拆分（如 7:1:2），禁止交叉使用
测试验证层面：必须包含 “训练集外样本”“跨场景样本”“低频次样本” 的泛化能力测试，泛化准确率衰减超出阈值则判定为过拟合
优化迭代层面：禁止仅针对测试用例调优（“应试式优化”），需聚焦通用能力提升，优化后需通过全新的盲测样本验证
上线监控层面：上线后需持续监控 Agent 在真实场景的表现，统计 “新场景识别率”“低频场景准确率”，及时发现过拟合风险并迭代

### 7.3 优化方案落地防控注意事项
模拟模式管控层面：优化过程中禁止构建任何形式的模拟测试/模拟模块/模拟环境作为核心优化手段，所有优化方案需基于真实企业级靶场环境、真实业务场景用例落地验证，确保优化效果可直接迁移至生产级场景，避免模拟环境与真实场景的效果偏差；
功能完整性保障层面：任何优化动作需以“不损失现有核心功能”为前提，优化方案设计阶段需前置梳理现有功能清单并做兼容性验证，禁止因单点优化导致其他功能模块失效、能力边界收缩；
回退方案约束层面：若优化后出现功能异常/效果不达预期，回退方案需满足“产出不减少、能力不降级”要求——回退仅恢复至优化前稳定状态，禁止通过删减测试用例、降低能力标准、缩减输出物范围等方式减少产出；回退执行前需完成影响面评估，明确回退后的能力达标率、产出物完整性需与优化前持平，且回退操作需保留完整日志便于追溯根因。

### 7.4. **优化落地安全性**：
优化方案禁止引入模拟模式，不得导致功能性缺失；回退方案需保障产出规模、能力标准不降低，避免因优化/回退操作造成测试流程中断或输出物不完整。
